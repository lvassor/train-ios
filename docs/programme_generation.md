# Programme Generation Architecture

**Version:** 5.0.0
**Last Updated:** March 2, 2026
**Status:** Production

## Overview

This document describes the complete architecture for train's programme generation system, from source data through to exercise selection in the iOS app.

---

## 1. File Architecture

### Source Files (database-management/)
| File | Purpose |
|------|---------|
| `equipment_prod.csv` | Equipment lookup table (IDs, categories, names, images) |
| `exercise_database_prod.csv` | Exercise data (220 exercises) |
| `exercise_instructions_prod.csv` | Multi-line exercise instructions (joined to exercises) |
| `exercise_contraindications_prod.csv` | Injury contraindications |
| `exercise_video_mapping_prod.csv` | Bunny Stream video GUIDs |
| `split_templates_prod.csv` | Split template source (120 rows) |
| `create_database_prod.py` | Database generator script |

### Generated/Runtime Files (TrainSwift/Resources/)
| File | Purpose |
|------|---------|
| `exercises.db` | SQLite database (generated by Python) |
| `constants.json` | Equipment/attachment mappings for Swift runtime |
| `split_templates.json` | Split configurations (generated from CSV) |

### Data Flow Pipeline
```
CSV Source Files (6 files)
    ‚Üì
Python Script (create_database_prod.py)
    ‚Üì
SQLite Database (exercises.db) ‚Üí TrainSwift/Resources/
    ‚Üì
```

## 2. Database Schema

### 2.1 equipment Table

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `equipment_id` | TEXT | PRIMARY KEY | Unique ID (e.g., "EP001") |
| `category` | TEXT | NOT NULL | High-level category (e.g., "Barbells", "Cables") |
| `name` | TEXT | NOT NULL | Specific name (e.g., "Squat Rack", "Straight Bar") |
| `image_filename` | TEXT | NULLABLE | Equipment image file (e.g., "00013101-Rack-Squat_equipment_max.png") |

### 2.2 exercises Table

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `exercise_id` | TEXT | PRIMARY KEY | Unique ID (e.g., "EX001") |
| `canonical_name` | TEXT | NOT NULL | Movement pattern grouping (e.g., "Bench Press") |
| `display_name` | TEXT | NOT NULL | User-facing name (e.g., "Barbell Flat Bench Press") |
| `equipment_id_1` | TEXT | NOT NULL, FK | Primary equipment ‚Äî references `equipment.equipment_id` |
| `equipment_id_2` | TEXT | NULLABLE, FK | Secondary/attachment equipment ‚Äî references `equipment.equipment_id` |
| `complexity_level` | TEXT | NOT NULL, CHECK('All','1','2') | Exercise complexity |
| `primary_muscle` | TEXT | NOT NULL | Main muscle targeted |
| `secondary_muscle` | TEXT | NULLABLE | Secondary muscle |
| `instructions` | TEXT | NULLABLE | Multi-line exercise instructions (joined from CSV) |
| `is_in_programme` | INTEGER | NOT NULL, DEFAULT 1 | Include in programme (0/1) |
| `canonical_rating` | INTEGER | NOT NULL, CHECK(0-100) | MCV ordering score (default 50) |
| `progression_id` | TEXT | NULLABLE | Comma-separated progression exercise IDs |
| `regression_id` | TEXT | NULLABLE | Comma-separated regression exercise IDs |

#### Complexity Level Meanings
- `"All"` = Available to everyone
- `"1"` = No experience / Beginner only
- `"2"` = Intermediate / Advanced only

#### Canonical Rating Guidelines
- 0-30: Isolation exercises (e.g., Bicep Curl)
- 40-60: Moderate compound (e.g., Romanian Deadlift)
- 80-100: Primary compound (e.g., Squat, Deadlift, Bench Press)

### 2.3 exercise_contraindications Table

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Row ID |
| `canonical_name` | TEXT | NOT NULL | Links to exercises.canonical_name |
| `injury_type` | TEXT | NOT NULL | Injury type (e.g., "Back", "Shoulders") |

**Unique constraint:** `(canonical_name, injury_type)`

### 2.4 exercise_videos Table

| Column | Type | Constraints | Description |
|--------|------|-------------|-------------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Row ID |
| `exercise_id` | TEXT | NOT NULL, UNIQUE, FK | Links to exercises.exercise_id |
| `supplier_id` | TEXT | NULLABLE | Video supplier ID |
| `filename` | TEXT | NOT NULL | Original filename |
| `bunny_guid` | TEXT | NOT NULL | Bunny Stream GUID (e.g., "620dd5e8-f79e-485a-89d6-ac941a2c2f5d") |

### 2.5 Foreign Key Relationships
```
equipment.equipment_id  ‚Üê‚îÄ‚îÄ  exercises.equipment_id_1  (NOT NULL, required)
equipment.equipment_id  ‚Üê‚îÄ‚îÄ  exercises.equipment_id_2  (NULLABLE, secondary)
exercises.exercise_id   ‚Üê‚îÄ‚îÄ  exercise_videos.exercise_id  (UNIQUE, 1:1)
exercises.canonical_name ‚Üê‚îÄ‚îÄ exercise_contraindications.canonical_name  (1:many)
```

### 2.6 Indexes

```sql
-- Equipment
idx_equipment_category ON equipment(category)

-- Exercise lookups
idx_exercises_canonical ON exercises(canonical_name)
idx_exercises_equip1 ON exercises(equipment_id_1)
idx_exercises_equip2 ON exercises(equipment_id_2)
idx_exercises_complexity ON exercises(complexity_level)
idx_exercises_muscle ON exercises(primary_muscle)
idx_exercises_programme ON exercises(is_in_programme)
idx_exercises_rating ON exercises(canonical_rating)

-- Contraindications
idx_contraindications_canonical ON exercise_contraindications(canonical_name)
idx_contraindications_injury ON exercise_contraindications(injury_type)

-- Videos
idx_videos_exercise_id ON exercise_videos(exercise_id)
idx_videos_bunny_guid ON exercise_videos(bunny_guid)
```

### 3.3 Equipment ID Resolution

After mapping questionnaire keys to category/name values, `ExerciseDatabaseManager.resolveEquipmentIds()` converts them to actual `equipment_id` values by querying the `equipment` table:

1. **Category base items**: For each category (e.g., "Barbells"), finds the equipment row where `name == category`
2. **Specific items**: Matches by equipment `name` (e.g., "Squat Rack")
3. **Attachment items**: Matches by equipment `name` (e.g., "Straight Bar", "Rope")

Returns a `Set<String>` of allowed equipment IDs used in the filtering chain.

## 4. App Flow: Questionnaire to Programme

### 4.1 Chronological Flow

```
1. User completes questionnaire
   ‚îú‚îÄ‚îÄ Equipment selection ‚Üí equipmentAvailable[]
   ‚îú‚îÄ‚îÄ Attachment selection ‚Üí detailedEquipment["attachments"]
   ‚îú‚îÄ‚îÄ Experience level ‚Üí experienceLevel
   ‚îú‚îÄ‚îÄ Goals ‚Üí primaryGoals[]
   ‚îú‚îÄ‚îÄ Target muscles ‚Üí targetMuscleGroups[]
   ‚îú‚îÄ‚îÄ Training days ‚Üí trainingDaysPerWeek
   ‚îî‚îÄ‚îÄ Split selection ‚Üí selectedSplit

2. DynamicProgramGenerator receives QuestionnaireData
   ‚îú‚îÄ‚îÄ Map equipment: questionnaire keys ‚Üí DB values
   ‚îú‚îÄ‚îÄ Map attachments: questionnaire keys ‚Üí DB values
   ‚îú‚îÄ‚îÄ Resolve equipment IDs via resolveEquipmentIds()
   ‚îú‚îÄ‚îÄ Check attachment warning (cables without attachments)
   ‚îî‚îÄ‚îÄ Determine split type from selectedSplit or daysPerWeek

3. Generate sessions for split
   ‚îú‚îÄ‚îÄ Load session templates from split_templates.json
   ‚îú‚îÄ‚îÄ For each session template:
   ‚îÇ   ‚îî‚îÄ‚îÄ For each muscle slot:
   ‚îÇ       ‚îî‚îÄ‚îÄ Select exercises using MCV algorithm

4. MCV Exercise Selection (per muscle slot)
   ‚îú‚îÄ‚îÄ Build user pool (filter by equipment IDs, complexity)
   ‚îú‚îÄ‚îÄ Filter by muscle group
   ‚îú‚îÄ‚îÄ Apply exclusions (already used IDs, display names, canonical names)
   ‚îú‚îÄ‚îÄ Select highest canonical_rating exercise
   ‚îî‚îÄ‚îÄ Remove from candidate pools

5. Assign sets/reps/rest based on goals

6. Return ProgramGenerationResult with warnings
```

### 4.2 Filtering Chain

```swift
// Stage 1: Database query filters
ExerciseDatabaseFilter {
    canonicalName: String?              // Filter by movement pattern
    equipmentCategories: [String]       // User's equipment categories
    equipmentSpecific: [String]         // User's specific equipment items
    attachmentSpecific: [String]        // User's attachments
    allowedEquipmentIds: Set<String>?   // Pre-resolved equipment IDs (overrides above)
    maxComplexity: Int                  // Based on experience (default 4)
    primaryMuscle: String?              // Target muscle
    excludeExerciseIds: Set<String>     // Already used
    onlyProgrammeExercises: Bool        // Filter by is_in_programme = 1 (default true)
}

// Stage 2: In-memory filters (MCV)
- excludedDisplayNames: Set<String>     // Programme-level (no exact repeats)
- excludedCanonicalNames: Set<String>   // Session-level (no variations)
```

**Note:** `excludeInjuries` exists on the filter but is NOT used for exercise filtering. Injuries produce UI warnings only.

### 4.3 Equipment Filtering SQL Logic

```sql
-- Exercises pass if:
-- 1. equipment_id_1 is in the user's resolved equipment IDs, AND
-- 2. equipment_id_2 is NULL (no secondary equipment needed), OR
--    equipment_id_2 is in the user's resolved equipment IDs

WHERE equipment_id_1 IN (:resolvedEquipmentIds)
  AND (equipment_id_2 IS NULL
       OR equipment_id_2 IN (:resolvedEquipmentIds))
```

## 5. MCV Heuristic Algorithm

### 5.1 Overview
MCV (Most Constrained Variable) ensures we don't get stuck by filling the hardest-to-fill slots first.

### 5.2 Selection Process

```
FOR each session template:
    WHILE slots remain unfilled:
        1. Build candidate lists for all unfilled slots
        2. Find slot with MINIMUM candidates (MCV)
        3. From that slot's candidates:
           - Select exercise with highest canonical_rating
        4. Mark selected:
           - Add canonical_name to session exclusions
           - Add display_name to programme exclusions
        5. Remove exercise from all candidate lists
```

### 5.3 Constraints

**Hard Constraints (never broken):**
- Equipment: user must have it (both equipment_id_1 and equipment_id_2)
- Complexity: user level must meet minimum
- No canonical_name repeat within session

**Soft Constraints (relaxed if needed):**
- No display_name repeat across sessions ‚Üí `exerciseRepeats` warning

### 5.4 Ordering
After selection, exercises are sorted by `canonical_rating` descending within each session. Compounds first, isolations last.

## 6. Sets, Reps & Rest Assignment

### 6.1 Rep Ranges (Goal-Based)

| Goal | Canonical Rating | Rep Range Options |
|------|-----------------|-------------------|
| Get Stronger only | > 75 | 5-8, 6-10 |
| Get Stronger only | ‚â§ 75 | 6-10, 8-12 |
| Increase Muscle | All | 6-10, 8-12 |
| Fat Loss | All | 8-12, 10-14 |
| Get Stronger + others | Uses Get Stronger rules |
| Fat Loss + Increase Muscle | Uses Fat Loss rules |

### 6.2 Sets
Always 3 sets for all exercises, all experience levels.

### 6.3 Rest Periods

| Canonical Rating | Rest (seconds) |
|-----------------|----------------|
| > 80 | 120 |
| 50-80 | 90 |
| < 50 | 60 |

## 7. Warnings & Edge Cases

### 7.1 Warning Types

| Warning | Trigger | User Message |
|---------|---------|--------------|
| `attachmentWarning` | Cables selected, no cable attachments | "You've selected Cable Machines but no cable attachments..." |
| `exerciseRepeats` | Soft constraint relaxed | "Some exercises repeated due to limited equipment..." |
| `noExercisesForMuscle` | Zero candidates for muscle | "No exercises available for [muscle]..." |
| `insufficientExercises` | Found < requested count | "Only found X of Y exercises..." |
| `lowFillRate` | Programme < 75% filled | "Programme only X% filled..." |

### 7.2 Attachment Warning Logic

```swift
if equipment.contains("cable_machines") {
    let cableAttachments = ["Straight Bar", "Rope", "D-Handles", "EZ-Bar Cable"]
    if !attachments.contains(where: cableAttachments.contains) {
        warnings.append(.attachmentWarning)
    }
}
```

## 8. Video Integration

### 8.1 Bunny Stream
- Video GUIDs stored in `exercise_videos.bunny_guid`
- Embed URL constructed via `BunnyConfig.videoEmbedURL(for: guid)`
- Format: `https://iframe.mediadelivery.net/embed/{libraryId}/{bunnyGuid}`
- Coverage: ~98% of exercises have video

### 8.2 Lookup Chain
```
ExerciseMediaMapping.videoGuid(for: exerciseId)
    ‚Üí ExerciseDatabaseManager.shared.videoGuid(for: exerciseId)
    ‚Üí O(1) cache lookup from videoGuidByExerciseId dictionary
    ‚Üí Returns bunny_guid string
    ‚Üí BunnyConfig.videoEmbedURL(for: guid) ‚Üí full embed URL
```

Video GUIDs are cached at startup in `ExerciseDatabaseManager` for O(1) access.

## 9. Injury/Contraindication Handling

**Important:** Injuries do NOT filter exercises from programmes.

Instead:
1. Exercises with contraindications are included in programme
2. UI displays warning icon for contraindicated exercises
3. User can still perform the exercise at their discretion

```swift
// WorkoutOverviewView checks at display time:
let contraindications = fetchContraindications(for: exercise)
let hasWarning = !Set(contraindications).isDisjoint(with: userInjuries)
// Show warning icon if hasWarning
```
